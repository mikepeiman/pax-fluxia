## Difference 1: Where Border Curves Are Computed

### Sacrosanct: Render-layer BĂ©zier
The renderer calls `quadraticCurveTo` at draw-time, computing smooth arcs from the Chaikin-smoothed control points. This **violates the architectural rule** that the renderer must not mutate or create geometry â€” it should only draw what the data layer provides.

### Master: Pre-computed geometry, straight rendering
The renderer calls `lineTo` on points that arrive from `pvv2MetricStage` (the geometry/compiler stage). This **respects the architectural rule**: all smoothing occurs in the data layer. The renderer is a pure draw function.

**However**, the Chaikin smoothing in `pvv2MetricStage` currently applies only to **border polylines**, not to fill polygons. This is the source of corner divergence.

**Future opportunity**: BĂ©zier densification (converting control points to a dense array of evaluated quadratic BĂ©zier midpoints) could be added as a **data-layer option** in `pvv2MetricStage`. It would need to handle outer edges and 3-way junctions properly to avoid gap artifacts. This is a feature to add, not something to restore from the sacrosanct commit.

## Difference 2: Fill/Border Alignment

> [!IMPORTANT]
> Neither commit achieves perfect fill/border alignment. This is a persistent issue across the project history.

### Sacrosanct: Chaikin applied in renderer to both fills and borders
`drawTerritoryFillWithHoles` applies `chaikinSmoothPolygon` to fill polygons at the same pass count as borders. This reduces divergence but does **not** eliminate it â€” sharp corners still appeared.

### Master: Fills use raw polygon points
`drawTerritoryFillOnly` draws `territory.points` directly â€” no smoothing. Meanwhile, `pvv2MetricStage` applies Chaikin to border polylines only.

**Root cause**: `pvv2MetricStage.ts` applies `chaikinSmoothPolyline` to the chained shared-edge polylines (contested borders between two owners), but the merged territory polygon points (used for fills) pass through without any smoothing.

**Fix**: Apply Chaikin smoothing to merged territory fill polygons inside `pvv2MetricStage` â€” this is a geometry-layer operation that respects the architecture. The renderer continues to draw without mutation.

## Difference 3: How Fills and Borders Coexist on PIXI.Graphics

### Sacrosanct: Two separate `PIXI.Graphics` objects
- `fillGraphics`: receives fill polygons (`.poly()` + `.fill()`)
- `borderGraphics`: receives border polylines (`.moveTo()` + `.quadraticCurveTo()` + `.stroke()`)

Both are children of `voronoiContainer`. Because `voronoiContainer` blanket-hides its children every frame, both need `visible = true` set at the start of each render call. Borders can be cleared independently of fills, and vice versa.

### Master: Single `PIXI.Graphics` object
Both fills and borders are drawn on `fillGraphics`:
- Fill polygons: `.poly()` + `.fill()`
- Border strokes: `.moveTo()` + `.lineTo()` + `.stroke()`
- `borderGraphics` exists but is immediately `.clear()`'d and `.visible = false`

The consequence: when the transition animation calls `fillGraphics.clear()` to redraw the frame, **both fills and borders are destroyed** and must be redrawn together every frame during transitions. In the sacrosanct commit, clearing `borderGraphics` for transition animation leaves the fill layer untouched.

## Difference 4: Transition-End Behavior

### Sacrosanct: No forced redraw
```typescript
if (rawT >= 1) {
    isSmoothTransitioning = false;
    prevSharedPolylines = null;
    // No redraw â€” PIXI.Graphics retains its drawn content between frames
    // Next frame: fingerprint check â†’ unchanged â†’ early return
}
```
`PIXI.Graphics` is a retained-mode display object â€” once drawn, its visual content persists until `.clear()` is called. The sacrosanct commit exploits this: after transition ends, the graphics retain whatever was last drawn, and the fingerprint check prevents unnecessary rebuilds.

### Master: Forced geometry-divergent redraw
```typescript
if (rawT >= 1) {
    // ... cleanup morphers ...
    fillGraphics.clear();
    // Redraw fills with drawTerritoryFillOnly (raw points, no smoothing)
    // Redraw borders with drawBorderPolylines (straight lineTo)
}
```
At the exact frame when `rawT >= 1`, the morpher's last rendered frame is destroyed and replaced by the steady-state drawing. If the morpher's geometry (resampled to 32 points, lerped, eased) differs from the steady-state geometry (Chaikin-smoothed polylines from `pvv2MetricStage`), there is a **visible snap** â€” a single-frame geometry discontinuity.

**Hypothesis**: This is the tick-based glitch the user observed.

## Difference 5: Transition Engine

### Sacrosanct: Single unified path
Both transition and steady-state call the same `drawBorderPolylines` function with the same rendering method. During transition, `buildLerpedPolylines` resamples prev/target to 32 points, lerps, then passes the result to `drawBorderPolylines` which applies Chaikin + BĂ©zier. At steady-state, pre-chained polylines go to the same `drawBorderPolylines`.

### Master: Mode-specific morpher classes
- `GraphicsPathMorpher`: does its own matching, resampling, lerping, easing, and drawing via `lineTo`
- `RopeBorderRenderer`: uses `PIXI.MeshRope` with animated control points â€” entirely separate texture-based rendering
- Legacy fallback: `buildLerpedPolylines` still exists but calls the modified straight-line `drawBorderPolylines`

Each morpher class produces geometry independently. The question is whether their geometry exactly converges to steady-state geometry at `t = 1.0`.

## Difference 6: Canonical Data Path

### Sacrosanct: Not present
The renderer computes everything inline from raw star positions via `d3-weighted-voronoi`.

### Master: FG2 canonical shell path added
```typescript
const canonicalShells = canonicalData?.shells ?? [];
if (canonicalShells.length > 0) {
    // Draw fills and borders from shell points
    return; // Skip legacy pipeline entirely
}
```
When `CanonicalTerritoryData` is provided with shells (from the FG2 pipeline), the renderer draws fills and borders from the same shell geometry and returns immediately. This bypasses the legacy `d3-weighted-voronoi` pipeline entirely.

This is the intended V3 architecture: geometry comes from one unified data source.

## Proposed Actions (Respecting Architectural Imperatives)

> [!IMPORTANT]
> All smoothing and geometry operations belong in the data/geometry layer (`pvv2MetricStage`). The renderer must not create or mutate geometry. These proposals comply with that requirement.

### 1. Fix fill/border divergence in `pvv2MetricStage`
Apply Chaikin smoothing to merged territory fill polygons in `pvv2MetricStage`, not just border polylines. This is a geometry-layer operation. **Non-destructive**: add the smoothing call; do not remove existing code.

### 2. Non-destructively test the transition-end snap hypothesis
In `PowerVoronoiRenderer.ts`, **comment out** (not delete) the forced redraw at transition-end (`rawT >= 1` block, lines 650â€“674 on master). Let the PIXI.Graphics retain the morpher's last frame. If the tick-based glitch disappears, the hypothesis is confirmed.

### 3. Evaluate region-protection for transitions
Instead of morphing all borders every frame, determine which stars changed ownership and morph only the frontiers those stars participate in. Unchanged regions keep their steady-state geometry untouched. This eliminates the transition/steady-state geometry divergence for stable regions.

**Critical evaluation**: This requires tracking per-frontier ownership, which `changedSiteIds` already provides. The morpher classes would need to accept a subset of polylines. The complexity risk: frontier segments shared between a changed and unchanged star â€” do they morph or not? Likely they must morph (one endpoint's ownership changed), but only on the frontier between the changed star and its new/old neighbor.

### 4. BĂ©zier densification as a data-layer option (future)
Implement `densifyBezierMidpoints` as an option in `pvv2MetricStage` â€” applied to both fill polygons and border polylines at the same time. Must handle 3-way junctions and outer edges to avoid gap artifacts. This is a new feature, not a reversion.
