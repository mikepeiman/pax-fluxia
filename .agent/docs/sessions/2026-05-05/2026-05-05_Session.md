## Challenged / Corrected Architectural Points

1. Shared geometry truth is still not truly unified across PVV4 and the field/perimeter families.
2. `GameCanvas` still manufactures transition truth for some families, which is an architectural violation.
3. The recorder/export stack starts after the first major normalization step, which hides the raw game graph from diagnostics.
4. The region-ID scheme is coupling identity to geometry output, which is structurally wrong for continuity and transition planning.
5. Region birth is invalid. Region collapse is only legitimate when the final star set of a region is conquered on that tick.
6. For complete multi-star region disappearance, the current default direction is per-star collapse, not whole-region collapse to a synthetic center.
7. `MSR` is the semantic name for the minimum star-range buffer constraint. `starMargin` is semantically wrong and should become `starWeight`. Lane-pair controls must be separated.
8. One shared ownership stage and one shared geometry stage must feed all render modes. Field families reconstructing prior truth locally is off-spec.
9. The geometry/topology foundation should explicitly treat the section between any two 3-way or world-edge junctions as the base structural section. Richer sub-segment detail can live inside that section, but the section itself is the right foundational unit.
10. DX should likely move from pure virtual-site nudging toward an explicit disconnect-zone representation that can be rendered and styled consistently across modes.
11. The recovery plan should gate motion work behind a locked conquest casebook and truth exports so the implementation does not drift back into ad hoc visual guessing.

## Centroid-ID Failure: Logical Consequence

A centroid cannot serve as region identity because identity must survive exactly the kinds of local boundary changes that move the centroid.

Why the current scheme guarantees failure:

1. Region identity is supposed to answer continuity:
   - is this the same owned region across frames?
2. A centroid is a function of the whole polygon:
   - any capture, split, merge, or border motion changes it
3. Therefore the ID changes when continuity is most important
4. That produces churn in:
   - region IDs
   - loop/region matching
   - transition continuity
   - diagnostics readability
5. Once identity is unstable, downstream code compensates heuristically and starts collapsing, respawning, or mispairing the wrong structures

## Better Direction

Region identity should come from gameplay/connectivity continuity, not geometric centroid coincidence.

The likely correct identity substrate is:

- owner ID
- connected star-component membership
- continuity matching against previous frame using star membership overlap and structural adjacency

The likely immediate deterministic key direction is:

- sorted owned-star IDs in the region
- serialized deterministically
- then hashed or string-keyed
- with continuity matching layered on top when merges/splits require richer reasoning

### DX contribution

User contribution:

1. examine every same-owner pair that is not lane-connected
2. test the midpoint ownership of the straight center-to-center span
3. if that midpoint still belongs to the same owner, require a disconnect zone
4. size that zone by:
   - depth as a percentage of star-to-star span
   - width as a percentage of adjacent lane spacing

Current evaluation:

- This is more explicit and controllable than the current DX virtual-site-only heuristic.
- The current code is already midpoint-oriented, but only in terms of enemy virtual-site placement.
- The missing step is explicit zone construction and mode-consistent rendering/styling.
- This should be added to the recovery plan as a later geometry-stage refinement, after shared truth is unified.
