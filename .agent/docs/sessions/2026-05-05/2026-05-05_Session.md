## Centroid-ID Failure: Logical Consequence

A centroid cannot serve as region identity because identity must survive exactly the kinds of local boundary changes that move the centroid.

Why the current scheme guarantees failure:

1. Region identity is supposed to answer continuity:
   - is this the same owned region across frames?
2. A centroid is a function of the whole polygon:
   - any capture, split, merge, or border motion changes it
3. Therefore the ID changes when continuity is most important
4. That produces churn in:
   - region IDs
   - loop/region matching
   - transition continuity
   - diagnostics readability
5. Once identity is unstable, downstream code compensates heuristically and starts collapsing, respawning, or mispairing the wrong structures

## Better Direction

Region identity should come from gameplay/connectivity continuity, not geometric centroid coincidence.

The likely correct identity substrate is:

- owner ID
- connected star-component membership
- continuity matching against previous frame using star membership overlap and structural adjacency

The likely immediate deterministic key direction is:

- sorted owned-star IDs in the region
- serialized deterministically
- then hashed or string-keyed
- with continuity matching layered on top when merges/splits require richer reasoning
