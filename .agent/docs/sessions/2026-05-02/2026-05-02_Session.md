## Addendum - inward-offset remnant-row and 24px snap correction

- The next live report sharpened the behavior further:
  - `Inward Offset` still left a row of tiny remnant squares along the frontier
  - the border looked correct through `1–23px`
  - at `24px` the visible result snapped because the old slider cap landed exactly on the next band boundary
- Important diagnosis:
  - the previous frontier-distance pass still used a purely per-cell clipping rule for every surviving band
  - with the live settings (`METABALL_GRID_SPACING_PX = 12`), this meant:
    - first ring near distance `0`
    - second ring near distance `12`
    - third ring near distance `24`
  - so the second ring could remain as a tiny clipped remnant until the slider hit `24`
  - then the next band reappeared, causing the observed snap
- Practical correction made in:
  - `pax-fluxia/src/lib/territory/frontier/distance.ts`
  - before per-side clipping, any band whose centerline is already inside the requested offset is now suppressed wholesale
  - only the current leading band can remain partially clipped
- Why this matters:
  - it removes the persistent high-offset remnant row
  - it removes the special failure at the old slider ceiling
  - it is still a surface-track change, not a timed-VFX hack
- UI contract change:
  - `TerritorySurfaceStyleTuning.svelte`
  - `Inward Offset` slider range widened from `24px` to `60px`

## Addendum - inward-offset repaint fallback bug

- The next live report was stronger than a tuning complaint:
  - the offset looked glitchy and unstable
  - the old `1-23px` behavior still appeared
  - the `24px` snap still appeared
  - higher values showed transient bands of gaps, but nothing stable
  - the user's key observation was: it looked like the result was being painted over
- That observation was correct.

### Problem restatement

- The new frontier-distance logic can return `null` visible square bounds for a cell when the requested offset fully suppresses that band/cell.
- In both active Metaball Grid families, the square fill loop still fell back to the legacy `drawFilledGridCell(...)` path when `squareBounds` was `null`.
- So fully suppressed cells were being repainted by the legacy square painter.
- That made the offset look unstable, intermittent, or capped even when the distance logic itself had suppressed the band correctly.

### Work done

- Updated:
  - `pax-fluxia/src/lib/territory/families/metaballGrid/MetaballGridPhaseEdgesFamily.ts`
  - `pax-fluxia/src/lib/territory/families/metaballGrid/MetaballGridFamily.ts`
- New behavior in both:
  - if the renderer is using `visibleSquareBoundsByGridIdx`
  - and the current square cell resolves to `null`
  - the loop now skips the cell entirely
  - it no longer falls back to `drawFilledGridCell(...)`

### Why this matters

- This is a real ownership bug:
  - the new offset path said "do not draw this cell"
  - the legacy fill path still said "draw it normally"
- The result was not a subtle VFX mismatch. It was an overwrite bug.
- It also explains why prior math-only fixes could appear to do nothing in the app.
