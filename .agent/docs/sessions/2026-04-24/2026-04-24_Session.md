## Facts, Assumptions, Unknowns

### Facts

- The user reported:
  - landing page loaded
  - clicking Start produced a blank screen
  - DOM elements still existed
  - the visible browser console only showed `[BuiltinMaps] Loaded 8 built-in maps`
- The Chrome `/.well-known/appspecific/com.chrome.devtools.json` request was informational and not causal.
- The browser/CDP diagnostic harness reproduced a runtime failure in the game-start path.

### Root cause

- The failing call path was:
  - `compileVectorGeometry()`
  - `UnifiedVectorGeometryMode.compute()`
  - `GeometryLayerCoordinator.compute()`
  - `TerritoryWorker.computeGeometrySync()`
  - `TerritoryRuntimeCoordinator.update()`
  - `GameCanvasTerritoryBridge.update()`
- The actual runtime error was:
  - `TypeError: input.tunables.entries is not a function`
- Contract review showed `tunables` is a plain object in the geometry-layer input, not a `Map`.

### Assumptions confirmed

- The blank screen was not the earlier `@pax/common/maps` export issue.
- The app shell was mounting; the failure was inside the territory-geometry path after Start.

## Implemented Changes

### 1. Start-game fix

File:

- `pax-fluxia/src/lib/territory/layers/geometry/compiler_UnifiedVectorGeometry.ts`

Change:

- replaced `Object.fromEntries(input.tunables.entries())`
- added a plain-object serializer for `input.tunables`

Why:

- aligns the compiler with the actual geometry input contract
- removes the start-game crash path

### 2. Input-path rect caching

File:

- `pax-fluxia/src/lib/components/game/GameCanvas.svelte`

Change:

- added cached canvas client-rect state
- added invalidation hooks on resize, scroll, and visual viewport changes
- replaced repeated `getBoundingClientRect()` usage in:
  - wheel zoom
  - benchmark pointer path
  - pointer down
  - long-press selection
  - pointer move drag path
  - pointer up
  - right click cancel
  - interaction overlay sizing

Why:

- the browser profile and previous session notes identified repeated layout reads as a real hot-path issue
- rect refresh cost is now visible and nearly absent from the benchmark hot path

### 3. Drag-preview overlay coalescing

File:

- `pax-fluxia/src/lib/components/game/GameCanvas.svelte`

Change:

- drag preview no longer forces an immediate overlay redraw on every raw `pointermove`
- drag-preview redraws now coalesce onto `requestAnimationFrame`
- immediate acknowledgement paths still use immediate rendering

Why:

- raw input streams can outpace frame cadence
- coalescing removes duplicate work while preserving interactive feel

### 4. Browser harness reliability

Files:

- `tools/debug/benchmark-browser-gameplay.ts`
- `tools/debug/diagnose-game-shell-load.ts`

Change:

- widened preferred port search range
- added ephemeral-port fallback when the preferred range is exhausted

Why:

- the benchmark harness failed locally because its original 50-port search window was too narrow for this machine
- harness reliability is part of the workflow requirement

### 5. Benchmark summary clarity

File:

- `tools/debug/summarize-browser-gameplay-benchmark.ts`

Change:

- prints explicit interaction-infrastructure measures even when they are too small to appear in the default top-12 focus list

Added explicit lines for:

- `game.input.clientRect.refresh`
- `game.input.dragPreview.present`

### Additional fixes

1. Removed the experimental interaction-isolation early-return path from `GameCanvas.svelte`
   - The experiment caused a real regression:
     - pending cancel visual acknowledgements were stranded
     - browser benchmarks showed `cancelVisualAckMs = null` and pointer-cancel results near `900ms`
   - The isolation branch was removed rather than carried forward in a broken state.

2. Fixed early-return render finalization in `GameCanvas.svelte`
   - The temporary helper introduced during the isolation experiment was invalid.
   - Early-return frames now finalize safely by:
     - presenting the interaction overlay if it has not already been presented
     - flushing pending visual acknowledgements
     - updating frame/FPS counters

3. Made queued order mutations visible to the overlay immediately
   - `getVisibleOrderTargetId(...)` now consults queued mutations first.
   - This makes optimistic cancel visibility reflect the queued cancel before store mutation completes.
   - Result: cancel visual acknowledgement returned to immediate behavior.

   - `tools/debug/summarize-browser-gameplay-benchmark.ts`

     - target handled -> visualAck

     - cancel handled -> visualAck

   - Removed the now-obsolete interaction-isolation summary line from the CLI report.
