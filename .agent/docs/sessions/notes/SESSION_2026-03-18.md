### 3. Territory Fill-Border Synchronization (Evening)
**Problem:** Fills and borders were two separate data paths — fills from `MergedTerritory` polygons (closed-polygon Chaikin), borders from `SharedPolyline` segments (open-polyline Chaikin). Different algorithms, different vertex sets, guaranteed divergence.

**Evolution of approaches:**
1. Independent `FillMorpher` — resampled fills separately → diverged from borders (reverted)
2. `assembleFrontierLoops` from morpher's interpolated polylines → failed (segment-reassembly doesn't produce valid closed loops from independently-morphed segments)
3. `FrontierLoopMorpher` on `MergedTerritory` closed polygons → fill-border binding confirmed clean, but whole territories rotated/morphed

**Fixes applied:**
- Dense resampling after Chaikin (`resampleClosedPolygonBySpacing`) — canonical frontier data at configurable pixel spacing
- Multi-region matching (arrays per owner + nearest-centroid matching) — fixed Map dropping duplicates for multi-region players
- Polygon rotation alignment (`alignClosedPolygon`) — minimizes total displacement to prevent whole-territory rotation

**Key files modified:** `pvv2MetricStage.ts`, `borderTransition.ts`, `PowerVoronoiRenderer.ts`, `game.config.ts`, `ControlsSection-Territory.svelte`

### 4. Architecture Decision: D-80

Canonical frontier data = array of arrays of x,y coordinate points with ownership. One data source for both fill and border rendering. Dense resampling produces the canonical vertices. `FrontierLoopMorpher` fills and strokes from same points.
