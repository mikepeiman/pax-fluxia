# Territory Shaping Constraints Audit - 2026-05-01

## Scope

Audit the current territory-shaping constraints:

- `MSR` = `MODIFIED_VORONOI_STAR_MARGIN`
- `CX` = corridor virtual sites / corridor extension
- `LP` = lane midpoint pairs (`TERRITORY_CX_CONTEST_*`)
- `DX` = disconnect gaps (`MODIFIED_VORONOI_DISCONNECT_*`, `TERRITORY_DX_WEIGHT`)

Audit dimensions:

1. intent
2. presence
3. mechanics of implementation
4. effectiveness

### Current status against spec

All four constraints are still present and still feed real geometry, but they are not equally faithful to intent.

- `MSR` is live, but overloaded and not cleanly equivalent to the intended lane-level exclusion rule
- `CX` is live and mechanically coherent in the modern path
- `LP` is live, but spacing is improperly coupled to `MSR`
- `DX` is live, but path normalization is inconsistent across geometry routes

The biggest risk is not absence. It is semantic drift and cross-path inconsistency.

## Constraint Audit

### 1. MSR

#### Intent

The intended meaning is **territory boundary breathing room around stars**. The frontier should not cut through a star or crowd it too tightly. This is first and foremost a player-owned territory boundary control.

Lane clearance is related but secondary:

- `Lane Margin` remains a distinct control
- when dedicated lane margin is disabled, `MSR` is intentionally the fallback lane margin value

#### Presence

- surfaced in `Frontier Topology` as `Minimum Star Margin`
- UI owner: `pax-fluxia/src/lib/components/ui/settings/ControlsSection-Territory.svelte`
- settings wiring owner: `pax-fluxia/src/lib/components/ui/settingsDefs.ts`
- default owner: `pax-fluxia/src/lib/config/territory.config.ts`

#### Mechanics of implementation

In the 0319 geometry compiler, `MSR` influences multiple behaviors:

- real-site weight and virtual-site weight scaling in the power Voronoi generation path
- contested midpoint pair spacing reuse
- an explicit post-merge vertex clamp via `pax-fluxia/src/lib/territory/geometry/minStarMargin.ts`

Separately, lane generation has its own lane-margin path in `pax-fluxia/src/lib/lanes/laneMargin.ts`. If dedicated lane margin is disabled, it intentionally falls back to `MSR`, which matches the current design intent.

#### Effectiveness

`MSR` is definitely active and visibly shapes geometry. Its primary territory-boundary purpose is real. The current weakness is not that it "fails to be lane margin." The weakness is that it is doing too many jobs:

- territory boundary stand-off around stars
- geometry weight influence
- fill clamp
- intentional lane-margin fallback when dedicated lane margin is disabled
- hidden lane-pair spacing coupling

So the corrected conclusion is:

- `MSR` is live and effective for territory/frontier shaping
- the lane-margin fallback behavior is correct and should stay
- the real problem is hidden cross-coupling to lane-pair spacing plus inconsistent application across different territory paths

### 2. CX

#### Intent

`CX` is corridor extension. Its purpose is to preserve same-owner territorial continuity and legibility along lanes by adding corridor-biased virtual influence.

#### Presence

- surfaced as `Corridor Virtual Sites (CX)`, `Corridor Sample Count`, `Corridor Weight`, and `Corridor Spacing`
- UI owner: `pax-fluxia/src/lib/components/ui/settings/ControlsSection-Territory.svelte`
- settings wiring owner: `pax-fluxia/src/lib/components/ui/settingsDefs.ts`
- default owner: `pax-fluxia/src/lib/config/territory.config.ts`

#### Mechanics of implementation

`CX` is generated by `pax-fluxia/src/lib/territory/corridor/buildCorridorVirtualSites.ts`.

Key behaviors:

- same-owner lanes get distributed corridor samples
- cross-owner lanes can also get distributed samples, but their central contested zone is intentionally skipped for that distributed pass
- the resulting virtual sites are injected into the 0319 power diagram through the geometry generator

#### Effectiveness

`CX` is the cleanest of the four constraints:

- it is present
- its mechanics align with its intent
- it materially affects modern canonical geometry

The only notable caveat is that legacy/sample-space renderer paths still exist elsewhere in the codebase, so not every renderer expresses `CX` identically. In the modern family-geometry path, it is good.

### 3. LP

#### Intent

`LP` here refers to contested-lane midpoint pairs:

- UI term: `Lane Midpoint Pairs`
- doc lineage: contested midpoint pair / contested-pair shaping

The goal is to keep contested-lane frontiers from collapsing into rigid or third-party-invaded shapes by planting owner-biased midpoint virtual pairs along cross-owner lanes.

#### Presence

- surfaced as `Lane Midpoint Pairs`, `Lane Midpoint Pair Count`, and `Lane Midpoint Pair Weight`
- UI owner: `pax-fluxia/src/lib/components/ui/settings/ControlsSection-Territory.svelte`
- settings wiring owner: `pax-fluxia/src/lib/components/ui/settingsDefs.ts`
- default owner: `pax-fluxia/src/lib/config/territory.config.ts`

#### Mechanics of implementation

`LP` is implemented inside `pax-fluxia/src/lib/territory/corridor/buildCorridorVirtualSites.ts` for cross-owner lanes:

- one left/right pair is created near the lane midpoint
- each pair member is attributed to one endpoint owner
- pair count replicates this pattern with shifts along the lane
- pair weight changes influence

Critical coupling:

- pair spacing does not own its own surfaced tuning
- it piggybacks on `MSR`, via the spacing that is routed through territory features into the corridor builder

#### Effectiveness

`LP` is real and useful, but the hidden `MSR` coupling weakens both tuning clarity and semantics. It works, but it does not own its own concept cleanly.

### 4. DX

#### Intent

`DX` is meant to prevent disconnected same-owner holdings from visually bridging into a single false territory blob by applying enemy-attributed disconnect pressure between components.

#### Presence

- surfaced as `Disconnect Gaps (DX)`, `Disconnect Weight`, and `Disconnect Distance`
- UI owner: `pax-fluxia/src/lib/components/ui/settings/ControlsSection-Territory.svelte`
- settings wiring owner: `pax-fluxia/src/lib/components/ui/settingsDefs.ts`
- default owner: `pax-fluxia/src/lib/config/territory.config.ts`

#### Mechanics of implementation

`DX` is built in `pax-fluxia/src/lib/territory/disconnect/buildDisconnectVirtualSites.ts`:

- same-owner connectivity is partitioned with union-find over the lane graph
- disconnected star pairs are identified
- for qualifying pairs, midpoint and perpendicular geometry are computed
- enemy-attributed virtual sites are placed on opposite sides to create separation pressure
- those sites are injected into the power diagram in the 0319 generator

#### Effectiveness

`DX` is a real geometry influence, but it is naturally conditional. It can legitimately no-op when the ownership graph does not present disconnected same-owner components worth separating.

The larger issue is inconsistency:

- config default weight is `3`
- UI slider max is `2`
- normalized family-geometry path clamps disconnect weight to `1`
- the legacy engine route can still pass the raw weight through

So `DX` is live, but it is not operating under a single stable contract.

## Cross-Cutting Findings

### 1. The constraints are not dead

All four constraints still feed real geometry in the 0319 pipeline and therefore still influence current rendered play, including the default `metaball_grid_phase_edges` family path.

### 2. The largest problems are semantic ownership and normalization consistency

The main defects are:

- `MSR` is overloaded
- `LP` is secretly spaced by `MSR`
- `DX` behaves differently depending on geometry route
- some legacy renderer/sample-space paths still duplicate or reinterpret shaping outside the canonical upstream contract

### 3. Presence is not the same as fidelity

Each constraint can be "present" in code and UI while still being wrong at the level of intended game meaning. That is the current situation with `MSR` most strongly, and with `DX` secondarily.

## Recommended Fix Order

1. **Normalize `DX` across paths**
   - one surfaced range
   - one internal range
   - one downstream interpretation

2. **Decouple `LP` spacing from `MSR`**
   - either own a dedicated surfaced spacing control
   - or explicitly document and embrace `MSR` coupling if that is truly intended

3. **Implement true lane-level `MSR` enforcement**
   - not just fill clamping and weight influence
   - actual "non-endpoint lanes stay outside this radius" semantics

4. **Reduce legacy path divergence**
   - ensure family/sample-space routes are consumers of canonical geometry truth rather than partial reinterpretations

## Bottom Line

- `MSR`: present, live, only partially true to intent
- `CX`: present, live, mechanically coherent, broadly effective in the modern path
- `LP`: present, live, effective, but conceptually coupled to `MSR`
- `DX`: present, live, conditionally effective, but contract consistency is weak
