# Session - 2026-05-01

## Focus
- Fix the map/viewport relationship in the preferred Phase Edges presentation so the map is centered and territory fills reach the viewport/grid-area boundaries.

## Facts
- The user observed a real asymmetry: fills reached the top and left bounds, but not the bottom and right.
- `GameCanvas.svelte` was fitting the world to a star-derived bounding box while territory renderers were also relying on `GAME_WIDTH` / `GAME_HEIGHT` as their world size.
- That is a bad architectural pattern: star extents are not the authoritative map rectangle.
- The first attempted fix on 2026-05-01 was rejected by live user verification:
  - it introduced margins on all four edges
  - the map still read as off-center
  - the user explicitly clarified that the star map / game map is not the same contract as the territory fill bounds
- The second-order architectural defect was that shifting the territory container without localizing the renderer inputs left the renderer in a mismatched coordinate space.
- The territory presentation request signature was also missing the presentation frame key, which meant a paused scene could reuse a stale fill render after the frame changed.

## Work Done
- Read `.agent/AGENT.md` again and confirmed the daily-doc/commit protocol before editing.
- Traced the active camera-fit path in `GameCanvas.svelte`:
  - `centerAndFit()`
  - `handleResize()`
  - `applyZoomTransform()`
  - `updateWorldBounds()`
- Confirmed that `updateWorldBounds()` was deriving the world from `min/max star positions + pad`, which is not the same contract as the generated map width/height.
- Added `pax-fluxia/src/lib/components/game/worldRect.ts` with `resolveViewportWorldRect()`:
  - prefers configured map extents when present
  - expands stale configured extents if live stars exceed them
  - falls back to star extents only when no configured map rectangle exists
  - keeps the world rooted at `(0,0)` so camera fit and territory rendering share one frame
- Updated `GameCanvas.svelte` to:
  - resolve the configured map rect for the current display orientation
  - center/fit against that rect
  - feed the same rect into `GAME_WIDTH` / `GAME_HEIGHT`
- Updated `gameStore.svelte.ts` so debug maps and saved maps also seed `_MAP_WIDTH`, `_MAP_HEIGHT`, `_MAP_PADDING_X`, `_MAP_PADDING_Y`.
- Added `worldRect.test.ts` for the resolver contract.
- Re-audited the rejected first fix and split the problem into three distinct rectangles:
  - star-fit camera rect
  - stable authored/display map rect
  - viewport-aligned territory presentation frame
- Added `pax-fluxia/src/lib/components/game/territoryPresentationSpace.ts`:
  - localizes display-space stars into the viewport-aligned territory frame
  - localizes canonical geometry into the same frame, including region/frontier/shell/topology coordinates
  - caches localized geometry by raw geometry reference + frame key
- Rewired `GameCanvas.svelte` so territory presentation now:
  - includes the territory frame key in the paused/live presentation signature
  - uses localized stars for direct territory renderers
  - uses localized stars and localized geometry for render-family inputs
  - uses localized stars/world size for canonical bridge and canonical controller presentation paths
- Added `territoryPresentationSpace.test.ts`.
- Wrote a dedicated post-mortem for the failed first viewport fix because the earlier implementation was architecturally wrong and user-corrected.
