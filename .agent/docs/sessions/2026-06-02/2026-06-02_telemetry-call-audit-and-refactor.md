# Telemetry Call Audit and Refactor

## Result

I treated `logPipelineStage` and `recordPerfEvent` call sites as removable telemetry by default. The active product and tool code no longer calls either API, but the executable API definitions are preserved: `logPipelineStage` remains in `pax-fluxia/src/lib/perf/pipelineTelemetry.ts`, and `recordPerfEvent` remains exported from `pax-fluxia/src/lib/perf/perfProbe.ts`. The removed call sites are preserved in the archive document.

Archive: [2026-06-02_logPipelineStage_recordPerfEvent_archive.md](./2026-06-02_logPipelineStage_recordPerfEvent_archive.md)

## Scope Reconciliation

The active project scan covered `pax-fluxia/src`, `tools`, `common/src`, `common/test`, and `common/tests`, excluding dependencies, generated output, build folders, the Git folder, harness output, and parallel worktrees.

- `logPipelineStage` active call sites archived: 82
- direct `recordPerfEvent` active call sites archived: 40
- `logPipelineStage` calls that also produced perf events through `perfEventName`: 66
- total event-producing call sites archived: 106
- distinct direct `recordPerfEvent` names/patterns: 39
- distinct routed `perfEventName` names: 65
- total distinct event names/patterns from both mechanisms: 104

The larger text-search count comes from imports, definitions, generated `.svelte-kit` output, old docs, and a parallel `.claude/worktrees` copy. The side-note `mapThumbnail.js` duplicate is generated output at `pax-fluxia/.svelte-kit/output/server/chunks/mapThumbnail.js`, not checked-in source. It is still worth excluding generated folders from human searches so generated chunks do not look like source duplication.

## Refactor Performed

- Removed all active `logPipelineStage(...)` call sites.
- Removed all active direct `recordPerfEvent(...)` call sites.
- Preserved the `logPipelineStage` API and its summary helpers in `pipelineTelemetry.ts` for possible future use.
- Preserved the `recordPerfEvent` API in `perfProbe.ts` for possible future use.
- Removed automatic browser observer event forwarding from `perfProbe.ts`; those were active event producers, not just reusable API definitions.
- Removed telemetry-only imports and summary helper imports from former call-site files.
- Left `measurePerf` and `recordPerfDuration` in place because they measure durations and still feed benchmark summaries.

## recordPerfEvent Assessment

These calls did not measure performance in the same way `measurePerf` does. They mostly wrote timestamped occurrence records with optional detail. Some details were useful, but the calls themselves did not make rendering faster, prove a frame was healthy, or guarantee a player-facing result.

Main consumer paths found:

| Consumer | What It Did | Judgment |
|---|---|---|
| `pax-fluxia/src/lib/perf/benchmarkBridge.ts` | Exposed the event list through `window.__PAX_BENCH__`. | Generic transport, not a reason for any specific event to exist. |
| `tools/debug/benchmark-browser-gameplay.ts` | Waited for exact input/order events and grouped pipeline/browser/input events by prefix. | Some exact waits had practical value, but they made the benchmark depend on hot-path event writes. |
| `tools/debug/diagnose-game-shell-load.ts` | Filtered exact render-yield and ship-defer events for one diagnostic action. | Narrow diagnostic use; should be rebuilt as a temporary probe when needed. |
| `tools/debug/summarize-browser-gameplay-benchmark.ts` | Printed event groups already captured by the benchmark script. | Reporter only; no independent diagnostic value. |

## Diagnostic Judgment

The event set was overbuilt for how it was actually used. There were 104 distinct event names or patterns, but most were not consumed by exact name. Broad prefix grouping gives counts and spans, but it does not answer a concrete debugging question unless someone is already running and reading the benchmark output.

The strongest events were the input/order exact waits in `benchmark-browser-gameplay.ts`. Those events were useful to that script, but they made product code carry benchmark synchronization markers. That is the wrong ownership. The benchmark should observe state changes through the benchmark bridge or install a temporary benchmark-only probe.

The weakest events were pipeline stage labels with long `purpose` fields. They looked like documentation, but at runtime they only emitted logs and occurrence records. They added code volume, naming burden, and argument construction in hot paths without evidence that people used the data.

## Remaining Dead or Questionable Code

- `tools/debug/benchmark-browser-gameplay.ts` still contains exact waits for event names that no source call now produces. That tool needs a follow-up refactor before order-latency scenarios can be trusted.
- `benchmarkBridge.ts` still exposes event-list APIs because `measurePerf` still records measure events. The exact-event parts are less valuable now and should be reviewed with the benchmark refactor.
- `GameCanvas.svelte` remains 8,029 LOC. It is still the primary cleanup target because it holds rendering, input, territory scheduling, bridge APIs, and diagnostic state in one file.
- `GameSettingsPanel.svelte` and multiple settings sections still produce many unused CSS warnings. The repeated `.log-*`, `.btn-*`, and `.var-row*` warnings are a visible smell of copied panel CSS and stale controls.
- Generated output and parallel worktrees create false positives during search. Searches should exclude `.svelte-kit` and `.claude/worktrees` unless the explicit goal is generated-output or worktree comparison.
- Several large render-family files are doing too many jobs. `MetaballGridPhaseEdgesFamily.ts`, `MetaballGridFamily.ts`, and `MetaballRenderer.ts` should be audited for planning/rendering/debug capture separation.

## Verification

- Search now finds only the preserved API definitions and bridge call: `recordPerfEvent` export in `perfProbe.ts`, `logPipelineStage` export in `pipelineTelemetry.ts`, and `logPipelineStage`'s internal call to `recordPerfEvent`.
- `bun run check`: exit code 0, 0 errors, 837 existing warnings.

## Recommendations

1. Keep the current removal. Do not reintroduce `logPipelineStage` or direct `recordPerfEvent` calls into gameplay or rendering paths.
2. Rewrite exact event waits in `benchmark-browser-gameplay.ts` to observe state or use benchmark-only probes.
3. Keep `measurePerf` for now because it records actual durations and has real benchmark consumption. Audit it separately once the event-waiting tool code is cleaned.
4. Run the next waste audit against `GameCanvas.svelte`, `benchmark-browser-gameplay.ts`, and settings-panel CSS. Those are the clearest follow-up targets from this pass.
