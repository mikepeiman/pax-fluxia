## Grid Gradient Transition Continuity And Star Glow

Purpose: keep the visible Grid Gradient fill transition, add star glow blend-fade on conquest, and remove animation repeats/restarts.

Plan/spec/status:

- Active plan: continue Grid Gradient conquest presentation polish.
- Governing architecture: `ownership -> geometry -> transition -> presentation`.
- Current status before work: Grid Gradient fill transition had visible dot movement; star glow snapped; transition sometimes restarted.
- Classification: off-spec implementation. Presentation timing and star visual color behavior contradicted the expected smooth conquest transition.

Trace findings:

- Star glow/ring rendering lives in `pax-fluxia/src/lib/renderers/StarRenderer.ts`.
- StarRenderer previously used `pendingConquests` to keep `effectiveOwner = previousOwner` until `transitionTime`, then deleted the pending state. That produced a hard color snap.

- Pending conquest previews in `buildRenderFamilyTransitionLifecycle()` used `startedAtMs = nowMs`.
- If the same pending conquest was seen on repeated pre-consume frames, or if a pending preview handed off to a handler-owned active entry, progress could restart from zero.

Implementation:

- Passed `pendingConquestStartedAtMsByKey` into `buildRenderFamilyTransitionLifecycle()`.
- Active entries now reuse the stable preview start when one exists for the same conquest key.
- Moved the frame's render-family transition state to a scope shared by territory and star presentation.
- Passed all active render-family transition sessions into `StarRenderer.ts` as per-star owner transitions.
- Added `resolveStarOwnerBlendVisual()` and changed star ring/glow drawing to cross-fade previous/new owner color passes.
- Extended pending conquest visual state in `VisualStateManager.ts` and `conquestHandler.ts` with `newOwner`, `startedAtMs`, and `durationMs` for fallback blend timing.

## Grid Gradient Transition Timing And Endpoint Alignment

User correction:

- Star glow fallback duration must match the conquest territory transition duration. It should not use Conquest `Color Delay` as a separate animation duration.
- The terminal fill-transition frame must match the settled NEXT dot field exactly; otherwise conquest appears to jank or pop at the end.

Reality trace:

- Territory transition duration is resolved by `resolveTerritoryTransitionDurationMs(effectiveTickMs)` in `pax-fluxia/src/lib/fx/handlers/territoryTransitionHandler.ts`.
- UI source for the duration is Timing -> Transition Clock -> Territory Transition (`TERRITORY_TRANSITION_MS`), with Timing -> Transition Clock -> Bind Territory Transition To Tick (`TERRITORY_TRANSITION_BIND_TO_TICK`) optionally clamping to `effectiveTickMs`.
- Conquest -> Resolution Timing -> Color Delay (`CONQUEST_COLOR_DELAY_TICKS`) is a tick-based delayed-color timestamp and is not the correct duration source for star glow as part of the conquest transition.
- Grid Gradient shader-field transition cells used transition-only minimum dot size, relaxed border offset, and side-specific seed even at full next-side alpha. That made the progress=1 frame differ from the later settled NEXT frame.

Implementation:

- Added `buildConquestStarOwnerPendingState()` to compute pending star owner visual state with `durationMs = resolveTerritoryTransitionDurationMs(effectiveTickMs)`.
- Updated `conquestHandler.ts` to store pending star owner state through that helper.
- Updated Grid Gradient CPU fallback sizing to resolve dot size per transition side and alpha.
- Updated shader-field fill rendering so transition endpoint sides render with settled/native sizing, border-offset suppression, and base noise seed.
- Added focused tests for star pending timing, transition endpoint sizing, shader endpoint behavior, and test config isolation.

Validation:

## Grid Gradient Performance Investigation

User report:

- Transition end looks smooth after the endpoint fix.
- Performance jank remains severe during Grid Gradient conquest.
- User requires major improvements only and no visual quality compromise.

Findings from screenshots and code trace:

- Chrome bottom-up screenshots show a `~448 ms` animation-frame/function-call spike.
- Hot path: `GridGradientFamily.update()` -> `resolvePlan()` -> `buildGridGradientPlan()` -> `buildGridClassification()` -> `resolveOwnerAt()`.
- `resolveOwnerAt()` dominates with about `302 ms` self time in the trace.
- Current Grid Gradient plan building is synchronous on the render-family update path.
- The closest better architecture is Metaball Grid's worker plan path using `metaballGridPlan.worker.ts` plus plan-pending visual progress handling.

Plan:

- Wrote `.agent/docs/plans/2026-06-12/GRID_GRADIENT_PERFORMANCE_MAJOR_FIX_PLAN_2026-06-12.md`.
- Major fix order: workerize current plan build, add plan diagnostics, replace per-cell point-in-polygon with grid raster classification, split steady owner-grid cache from transition diff, then feed shader packing from typed plan data.

Implementation:

- Added `typedClassification.ts` for Grid Gradient typed owner/role arrays.
- Added a scanline raster classifier for non-jittered square/hex grids, with jittered grids still using the existing point classifier path.
- Split steady owner-grid caching from transition role diffing.
- Added `gridGradientPlan.worker.ts` and worker request/response types.
- Changed `GridGradientFamily.resolvePlan()` to render the cached plan while changed plans build in a Worker, then commit the worker plan and use the original transition start/duration for local visual progress.
- Kept synchronous build only for the initial/no-Worker fallback path.
- Changed shader-field texture packing to consume typed owner, role, and flip-time arrays directly when available.
- Added diagnostics for classifier path, owner-grid cache hits, plan worker scheduled/wait/commit, requested plan pending state, and plan build splits.
