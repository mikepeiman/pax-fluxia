## Lane truth ownership

- User reported that lanes still disappear visually above a certain `Lane Margin` value while movement and attacks remain mechanically connected.
- User explicitly required that visible lane manifestation be derived only from map truth, not from a separate render-time interpretation layer.
- Tracing confirmed `/common` already carries lane-path truth through shared types and schema.

## Fixed-map lane audit and hierarchy

- Added a frozen-map audit tool:
  - `bun run debug:lane-audit -- --saved-map common/resources/saved-maps/inner_circle_apr_13.json`
- Working clearance definition for now:
  - center of non-endpoint star to lane centerline
- Shared geometry remap now uses the explicit blocker-driven rule:
  - find the nearest blocking star-to-lane witness
  - insert a vertex on that exact shortest path
  - push it to Lane Margin and not beyond it
  - repeat deterministically if another blocker still violates the constraint
- Follow-up correction:
  - the first deterministic version over-pruned and nearly eliminated curved lanes
  - fixed by removing fail-fast multi-blocker rejection
  - added an explicit clearance epsilon for numeric stability
  - replaced aggressive curved smoothing with conservative deterministic corner-rounding
- Deterministic result from the frozen map:
  - false-positive curves were real and are now gone
  - above roughly `LM 175`, the strict feasible graph is disconnected
- This means high-LM behavior is no longer being treated as a guess or renderer problem.
- The geometry hierarchy is now explicit:
  1. full traversal connectivity is the winning constraint
  2. keep straight when the chord satisfies LM
  3. if the chord violates LM:
     - remap enabled: use a satisfying adjusted path
     - remap disabled: reject that lane and seek replacement elsewhere
  4. if the strict feasible graph is still disconnected, add an explicit graph-level connectivity-restoration edge

## Hex fixture correction and square-map cliff

- Removed all embedded ruler fixtures from `common/resources/saved-maps/lane_margin_hex_layers_2p.json`.
  - the previous fixtures were not consistently measuring the relevant nearby lane body
  - they were misleading as a lane-margin diagnostic
- Traced the square-map `LM > 200` cliff in shared geometry:
  - the dramatic step was not primarily a connectivity issue
  - it was caused by lane-margin measurement treating the exact lane endpoints as part of the constraint
  - nearby non-endpoint stars can therefore create a hard upper bound even when the body of the lane still has room to bend
- Added an endpoint guard in `common/src/mapgen/lanePolylines.ts` so lane-margin solving measures the body of the lane instead of collapsing at the exact start/end point.
- Updated `tools/debug/audit-lane-constraints.ts` to use the same endpoint-guard distance semantics, so audit counts now match the shared solver.
