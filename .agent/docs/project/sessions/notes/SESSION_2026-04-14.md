# Session Notes - 2026-04-14

## Perimeter Field non-negotiables

- We are only working on `perimeter_field` unless the user explicitly says otherwise.
- Transition diagnostics must be captured from the real live gameplay render path, not from any reconstructed or side-effecting replay renderer.
- `PREV` means the actual gameplay frame/state before conquest. `NEXT` means the actual gameplay frame/state after conquest. Diagnostics are not allowed to synthesize substitutes and call them `PREV` or `NEXT`.
- `perimeter_field` uses perimeter vstars as the display ownership primitive. Transition must be correspondence and motion between real `PREV` and `NEXT` perimeter-vstar states, not a substitute temporary-sample model.
- Affected `PREV` static perimeter samples must be replaced during conquest, not double-counted with transition samples.
- Vstars in diagnostics must be clearly owner-colored and clearly labeled:
  - `S#` static
  - `T#` target/next static
  - `O#` loser / previous-owner transition
  - `N#` victor / new-owner transition
- Diagnostic scrub must step exact captured conquest frames, not normalized `0-100` progress.
- Export bundles must contain rendered gameplay frames plus diagnostic overlays, and `diagnostic.json` must stay minimal and readable.
- Panel subsection state is expected to persist like the rest of the settings UI.
- Commit the full worktree on each requested checkpoint, not selective slices.
- Change vstars must be determined from changed active fronts / contested topology, not star-centered heuristics.
- Some motion paths may need non-linear routing to avoid crossing unrelated frontiers.

## Current corrective direction

- Preserve upstream geometry identity. Do not drop `starIds` when adapting geometry into render-family input.
- Deterministic star-to-region membership is necessary but not sufficient; changed-front / section identity still has to drive moving-sample selection.
- Remove synthetic family-side debug replay paths when live captured frames already exist.
- Keep `DX` defined as midpoint-based virtual-star logic between disconnected same-owner stars.

## Known current deviations

- The current implementation still uses source-level replacement and heuristic correspondence in places where the mode design requires changed-front-driven correspondence.
- The current export path still conflates clean gameplay frames with debug overlays in some artifacts.
- The current `PREV` capture path remains suspect until transition-start capture is explicit and atomic.

## User-reported failure signals to treat as authoritative

- If the user says `PREV` is wrong, assume the diagnostic path is wrong until traced.
- If play and paused show different vstar states, assume diagnostics are mutating or diverging from live render state.
- If transition disappears after a diagnostics change, assume the diagnostics wiring touched live render state and trace from that boundary.
