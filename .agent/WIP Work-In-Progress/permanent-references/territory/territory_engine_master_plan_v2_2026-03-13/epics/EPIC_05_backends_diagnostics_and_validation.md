# EPIC 05: Backends, Diagnostics, And Validation

## Goal

Drive backend parity, clarify backend responsibilities, and make route-truth and visual validation mandatory parts of territory execution work.

## Included Tasks

- `BKD-001` through `BKD-005`
- `DBG-001` through `DBG-004`

## Dependencies

- `EPIC_01_foundation_and_contracts`
- at least one demo-ready native path from `EPIC_02_static_methods`
- dynamic and hybrid tasks as they become demo-ready

## Execution Order

1. Lock the backend contracts and parity expectations.
2. Expand trace and step planning to show backend-aware route truth.
3. Standardize browser, screenshot, and benchmark validation.
4. Use parity comparison to keep backends honest as more native methods come online.

## Done Criteria

This epic is done when:

- `PVV2`, `PVV3`, and `DF` have explicit parity expectations
- browser and screenshot validation is part of the standard workflow
- demo-ready routes can be compared across backends with saved evidence
