# DY4 Renderer Comparison: Good Commit vs Master

**Good commit**: `78c5372` — "canonize DY4 Optimal Transport as sacrosanct default border mode (D-77)"
**Current master**: `9f676c1` — HEAD as of 2026-03-18

---

## Architecture Overview

| Aspect | Good (`78c5372`) | Master (`9f676c1`) |
|---|---|---|
| File size | 1427 lines, self-contained | 939 lines + 1501 in `geometry/` |
| Geometry | All inline (voronoi, merge, edge extract, Chaikin) | Extracted to `pvv2MetricStage.ts` |

## Critical Difference 1: Border Drawing Math

This is the single most impactful change.

### Good: Bézier curves through midpoints ⭐
```typescript
// drawBorderPolylines @ line 504-549 (good commit)
if (pts.length === 2) {
    graphics.moveTo(pts[0][0], pts[0][1]);
    graphics.lineTo(pts[1][0], pts[1][1]);
} else {
    // Quadratic Bézier through midpoints for smooth arc geometry
    graphics.moveTo(pts[0][0], pts[0][1]);
    const mid0x = (pts[0][0] + pts[1][0]) / 2;
    const mid0y = (pts[0][1] + pts[1][1]) / 2;
    graphics.lineTo(mid0x, mid0y);
    for (let i = 1; i < pts.length - 1; i++) {
        const midX = (pts[i][0] + pts[i + 1][0]) / 2;
        const midY = (pts[i][1] + pts[i + 1][1]) / 2;
        graphics.quadraticCurveTo(pts[i][0], pts[i][1], midX, midY);
    }
    const last = pts[pts.length - 1];
    graphics.lineTo(last[0], last[1]);
}
```
**Effect**: Smooth curves. Each control point generates a smooth arc. With Chaikin pre-smoothing (3 passes default), the result is very smooth borders.

### Master: Straight lines only
```typescript
// drawBorderPolylines @ line 271-294 (master)
graphics.moveTo(pts[0][0], pts[0][1]);
for (let i = 1; i < pts.length; i++) {
    graphics.lineTo(pts[i][0], pts[i][1]);
}
```
**Effect**: Polygonal borders. Each segment is a straight line. The rationale was "curves are baked into the point data" via `densifyBezierMidpoints` — but this function is only used in a limited context and the smoothing quality is different.

### Impact
- Fixed Bézier math naturally produces smooth borders from relatively few control points
- Straight-line-only requires many more pre-densified points to appear smooth, and any resampling (32 pts for transitions) shows visible faceting

## Critical Difference 2: Fill Smoothing

### Good: Chaikin smoothing applied to fills
```typescript
// drawTerritoryFillWithHoles @ line 862-885 (good commit)
const fillPts = smoothPasses > 0
    ? chaikinSmoothPolygon(territory.points, smoothPasses)
    : territory.points;
graphics.poly(fillPts.flat());
graphics.fill({ color: territory.color, alpha });
```
**Fills AND borders both apply Chaikin** → they share the same curved geometry → zero visual divergence at edges.

### Master: Raw polygon points for fills
```typescript
// drawTerritoryFillOnly @ line 446-469 (master)
// Use raw polygon points — NO chaikin here. Stage owns smoothing.
graphics.poly(territory.points.flat());
graphics.fill({ color: territory.color, alpha });
```
The comment says "stage owns smoothing" but the stage (pvv2MetricStage) applies Chaikin to **border polylines only**, not to fill polygons. This creates the B-42 divergence again: **fills have sharp corners, borders are smooth**.

> [!WARNING]
> This is a regression of the B-42 fix. The good commit solved B-42 by applying Chaikin to both fills and borders at the same pass count. Master removed fill smoothing, re-introducing the divergence.

## Critical Difference 3: Transition Architecture

### Good: Unified draw path
```
conquest → snapshot prev polylines → each frame:
  1. buildLerpedPolylines(prev, target, easedT)
     - resample both to 32 points
     - match by ownerPairKey + centroid proximity
     - lerp positions
  2. drawBorderPolylines(lerped, smoothPasses, ...)
     - Chaikin smooth the lerped points
     - Draw via quadraticCurveTo
  Result: Transition AND steady-state both use identical drawBorderPolylines
```

### Master: Dual morpher system
```
conquest → snapshot prev polylines → each frame:
  IF pixi_graphics_morph:
    1. GraphicsPathMorpher.drawFrame(g, rawT, width, alpha)
       - has own matching/resampling/lerping/easing
       - draws with lineTo (straight lines)
  ELIF pixi_mesh_rope:
    1. RopeBorderRenderer.update(rawT, alpha)
       - PIXI MeshRope with animated control points
       - separate texture-based rendering
  ELSE (legacy fallback):
    1. buildLerpedPolylines (same as good, but calls modified drawBorderPolylines)
       - draws with lineTo (no Bézier)
```

**The fundamental problem**: In the good commit, transitions and steady-state both call the same `drawBorderPolylines` with the same Bézier math. In master, the morpher classes have their own independent drawing logic that differs from the steady-state path. The transition-to-steady-state handoff produces a visible snap.

## Critical Difference 4: Graphics Layer Separation

### Good: Two layers
```typescript
// Good commit
if (fillGraphics) fillGraphics.visible = true;
if (borderGraphics) borderGraphics.visible = true;
// ...
// Fills drawn on fillGraphics
// Borders drawn on borderGraphics (separate)
```

### Master: Single layer
```typescript
// Master
if (fillGraphics) fillGraphics.visible = true;
// borderGraphics stays hidden — borders are now strokes on the fill path
// ...
// Fills AND borders drawn on fillGraphics
if (borderGraphics) {
    borderGraphics.clear();
    borderGraphics.visible = false;
}
```

**Impact**: In the good commit, borders can be drawn/cleared independently without disturbing fills. In master, clearing fillGraphics for transition animation also clears the borders, requiring them to be redrawn every frame.

## Critical Difference 5: Transition Completion Handoff

### Good: Clean completion
```typescript
// Good commit @ line 1027-1031
if (rawT >= 1) {
    isSmoothTransitioning = false;
    prevSharedPolylines = null;
    // Next frame: fingerprints unchanged → no rebuild → uses cached fills/borders
}
```
After transition completes, the existing fillGraphics and borderGraphics retain their last-drawn content. No redraw needed.

### Master: Forced redraw at completion
```typescript
// Master @ line 650-674
if (rawT >= 1) {
    isSmoothTransitioning = false;
    // ... cleanup morphers ...
    // Redraw steady-state fills + borders directly on fillGraphics
    fillGraphics.clear();
    for (let i = 0; i < lastMergedTerritories.length; i++) {
        drawTerritoryFillOnly(fillGraphics, lastMergedTerritories[i], ...);
    }
    if (targetSharedPolylines && ...) {
        drawBorderPolylines(fillGraphics, targetSharedPolylines, 0, ...);
    }
}
```
The forced redraw at transition-end uses `drawTerritoryFillOnly` (raw points) + `drawBorderPolylines` (straight lines), which differs from what the morpher was drawing. **This is the tick-based snap**.

## Critical Difference 6: Geometry Pipeline Location

### Good: Inline in renderer
All geometry (site building, d3-weighted-voronoi, cell merge, edge extraction, Chaikin) computed inline in `renderPowerVoronoi`. The renderer fully controls smoothing.

### Master: Extracted to pvv2MetricStage
```typescript
const stageResult = executePVV2MetricStage(stars, connections, stageConfig);
const { cells, mergedTerritories, sharedEdges, sharedPolylines, worldBorderPolylines, enclaveMap } = stageResult;
```
The stage returns pre-smoothed polylines (Chaikin applied to border polylines in the stage), but raw fill polygons. The renderer no longer controls smoothing — it just receives data.

**This is architecturally cleaner** but introduced the fill/border divergence because the stage smooths borders but not fills.

---

## Summary of Regression Causes

| Symptom | Root cause | Fix approach |
|---|---|---|
| Borders look faceted/angular | Bézier `quadraticCurveTo` removed, replaced with straight `lineTo` | Restore Bézier drawing in `drawBorderPolylines` |
| Fill/border corner divergence (B-42 regression) | Fill smoothing removed from renderer, stage only smooths borders | Either smooth fills in stage too, or restore Chaikin in renderer |
| Tick-based glitch at transition end | Morpher draws different geometry than steady-state redraw | Unify: both paths use same draw function |
| Transition-to-steady-state snap | Forced redraw at rawT>=1 uses different code path | Keep morpher's last frame visible, don't force redraw |
| MeshRope visibility flicker | blanket-hide + re-show cycle per frame | Good commit didn't have MeshRope at all |

## Recommended Fix Order

1. **Restore Bézier drawing** in `drawBorderPolylines` — single most impactful change
2. **Restore fill Chaikin** — either in pvv2MetricStage (smooth fills too) or in renderer
3. **Unify transition completion** — don't force-redraw, let last morpher frame persist
4. **Or: restore two-layer separation** — fills on fillGraphics, borders on borderGraphics
