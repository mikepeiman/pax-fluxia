### 4. CRITICAL BUG: Map Deduplication in Production Transitions
**Root cause**: All frontier matching code used `Map<ownerPairKey, singleValue>`. When two territories share multiple disjoint border segments (same owner pair key), the Map only kept the LAST segment. Roughly half of all frontier segments were being silently dropped.

**Impact**: Dropped segments received ZERO transition morphing — they would snap instantly during conquests instead of smoothly interpolating via optimal transport. This explains persistent transition glitches that were never properly attributed.

**Affected files**:
- `GeometryTopologyDiff.ts` — `diffPolylines()` L77-79
- `interpolatePolylines.ts` — `matchPolylinesByKey()` L166-173  
- `CorrespondencePlanner.ts` — `planFrontierCorrespondence()` L13-18
- `FrontierTopologyBuilder.ts` — `buildSharedFrontierMap()` L39-43 (WARNING added, contract-typed)

**Fix**: All three transition files now use multimap arrays (`Map<string, T[]>`) and match segments by index within each ownerPairKey bucket.

See: `POST_MORTEM_2026-03-24_FRONTIER_DEDUP.md`
